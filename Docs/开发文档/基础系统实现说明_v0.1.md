# 《异界旅人》基础系统实现说明 v0.1

## 1. 本次目标

本次根据《异界旅人》策划案 v0.2，先实现不依赖具体场景和 UI 的基础领域代码，覆盖：

- 行动驱动的全局连续时间。
- 容器内仍以卡牌表达物品，同类物品卡按数量合并。
- 全局标记、数值和地点运行状态。
- 可由 Unity Test Runner 执行的 EditMode 单元测试。

本次没有替换 StackCraft 现有的 `TimeManager`、`CombatManager`、`CardManager` 或存档结构，也没有制作世界地图、容器 UI 和 NPC 对话界面。这样可以在不破坏模板闭环的前提下，先稳定策划案依赖的数据规则。

## 3. 行动驱动连续时间

核心类：`CardColony.TimeSystem.ActionDrivenWorldClock`

### 已实现规则

- 空闲且未主动等待时，调用 `Tick` 不会推进时间。
- 工作、采集、制作、探索、旅行或战斗开始时，通过 `BeginAction` 注册耗时行动。
- 至少存在一个活动行动时，世界时间连续推进。
- 行动句柄释放后停止计时；多个并行动作共享同一个时钟，不重复叠加时间。
- 玩家可通过 `IsWaiting` 主动让空闲时间流逝。
- `IsPaused` 的优先级高于行动和主动等待。
- 支持常速和四倍快速。
- `DayNumber` 与 `MinuteOfDay` 从累计世界分钟自动计算，能够连续跨日。
- 时钟倍率、累计时间和帧增量会拒绝未定义倍率及 `NaN`/无穷大，避免坏数据污染存档。

### 预期接入方式

```csharp
IDisposable action = worldClock.BeginAction("gather-herbs-001");

// 由未来的 WorldClockDriver 每帧调用。
worldClock.Tick(Time.unscaledDeltaTime);

// 采集完成、取消或对象销毁时必须释放。
action.Dispose();
```

未来接入现有系统时，应分别在制作任务、旅行、探索和战斗开始/结束处持有并释放行动句柄。纯 UI 操作、对话和容器整理不注册行动，因此不消耗世界时间。

## 4. 卡牌化容器

核心类：

- `ItemCardStack`：容器里的一张可见物品卡。
- `CardContainer`：受槽位和重量约束的卡牌容器。
- `InventoryAddResult`：放入容器后的接受与拒绝数量。

### 物品卡字段

- `ItemId`：稳定物品定义 ID。
- `InstanceId`：这张容器卡的唯一实例 ID。
- `Quantity`：卡面数量。
- `Quality`：品质。
- `BatchId`：商品批次。
- `MaxStackSize`：单张卡最大数量。
- `UnitWeight`：单件重量。

### 合并规则

只有物品 ID、品质、批次、堆叠上限和单位重量精确一致时，物品才会合并到同一张卡。数量超过单卡上限时，会占用新的卡牌槽位。不同品质、批次或物理参数的商品始终保留为独立卡牌，避免近似重量在大量合并后突破容器承重。

`CardContainer.Add` 会同时考虑：

- 已有卡牌的剩余堆叠空间。
- 剩余槽位。
- 剩余重量。

只有单位重量严格为 `0` 的物品不占承重；任何正数重量（包括极小值）都会参与容量计算。

无法放入的部分通过 `RejectedQuantity` 返回，不会静默丢失。`TryRemove` 支持从一张数量卡中拆出指定数量，并生成新的卡牌实例 ID，可用于拖出背包、交付任务或放入另一个容器。

### UI 约定

未来的背包、马车和仓库界面应直接遍历 `CardContainer.Cards` 生成卡牌，不应把内容渲染成普通文字列表。`Quantity`、`Quality` 和 `BatchId` 应显示在卡面或详情面板中。

## 5. 世界与地点状态

核心类：`CardColony.World.WorldStateStore`

### 已实现状态

- 布尔标记：任务选择、魔王削弱、世界事件是否发生等。
- 整数数值：NPC 关系、地区声望、准备度等。
- 地点状态：是否发现、探索进度、已清理遭遇 ID。
- `WorldStateSnapshot`：带版本号的深拷贝快照，可作为后续存档数据的一部分。

地点状态以稳定的 `LocationId` 为键，不依赖 Unity 场景名称。因此河湾村、低语森林等地点可以共享同一套通用地点场景，同时分别保存探索和事件状态。

## 6. 测试过程与覆盖

本次按照 RED-GREEN 流程开发：

1. 先编写测试。
2. 确认测试因 `TimeSystem`、`Inventory`、`World` 功能尚不存在而编译失败。
3. 编写满足测试的最小实现。
4. 运行 24 项测试，覆盖空闲/行动/等待/暂停/倍速/并行动作/跨日、非法数值、卡牌合并/拆分/槽位/普通与极小重量/超大可容纳数量，以及世界状态快照恢复。

Unity 编辑器中可通过 `Window > General > Test Runner > EditMode` 运行 `CardColony.Core.Tests`。测试代码不依赖场景和预制体。

## 7. 下一步接入建议

建议按以下顺序继续：

1. 新增 `WorldClockDriver`，在 Unity 生命周期中驱动 `ActionDrivenWorldClock`，并显示日期与时间。
2. 为 `CraftingTask`、旅行、探索和战斗添加耗时行动适配器。
3. 新增容器卡牌 UI，将 `ItemCardStack` 映射到现有卡牌预制体。
4. 为现有制作、任务和交易增加统一物品查询/扣除接口，使桌面卡和容器卡都能作为物品来源。
5. 把累计世界时间、`WorldStateSnapshot` 和容器内容接入带版本号的新存档结构。
6. 再实现 `LocationDefinition`、世界地图和通用地点场景。

## 8. 已知边界

- 当前容器模型尚未实现两个容器之间的事务式转移；UI 接入前应补充“目标拒绝时完整回滚”的测试。
- 当前时钟的快速速度固定为四倍，后续可根据 UI 体验调整，但不建议让各系统直接修改倍率。
- 当前快照只覆盖世界和地点状态；角色、队伍、容器与时钟应在统一的 `RunSnapshot` 中组合。
- 当前实现没有修改自动战斗逻辑，符合策划案 v0.2“首版保持现有实时自动战斗”的决定。
